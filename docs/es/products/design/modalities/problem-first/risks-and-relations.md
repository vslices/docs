# Riesgos y *tradeoffs*

_Problem-First_ es útil, pero puede fallar cuando el equipo trata la declaración inicial del problema como verdad completa. Riesgos comunes incluyen:

* optimización local
* resolución de síntomas
* contexto circundante débil
* dependencias ocultas
* análisis de riesgo incompleto
* criterios de éxito poco claros
* sobreajustar la solución a un caso visible
* ignorar responsabilidades afectadas
* construir demasiado rápido desde una etiqueta de problema
* convertir un problema de negocio en una tarea técnica demasiado pronto

Para evitar estos riesgos, el equipo debería mantener el problema conectado a su contexto. El equipo debería preguntar:

* ¿Entendemos el problema o solo su etiqueta?
* ¿Estamos resolviendo una causa o un síntoma?
* ¿Qué flujo circundante podría cambiar la decisión?
* ¿Qué riesgos estamos aceptando?
* ¿Qué probaría que el problema mejoró?
* ¿Estamos manteniendo la solución proporcional al problema?
* ¿Estamos documentando porque ayuda o porque se siente seguro?
* ¿Tenemos suficiente contexto para construir responsablemente?

_Problem-First_ debería crear progreso enfocado. No debería convertirse en un atajo para entendimiento superficial.

# Relación con otras modalidades

_Problem-First_ no siempre es la mejor modalidad. Es mejor cuando hay un problema claro y se puede reunir suficiente contexto alrededor de él.

* Un equipo puede empezar con Context-First para entender un dominio amplio, y luego moverse a _Problem-First_ cuando un problema valioso se vuelve claro.
* Un equipo también puede moverse desde _Problem-First_ a Slice-First cuando la mejor forma de validar la mejora es construir rápidamente una pequeña *vertical slice*.

Las modalidades pueden combinarse a través de iteraciones. No son identidades, son estrategias.
