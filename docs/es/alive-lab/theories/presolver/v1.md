# PResolver una metodología para la resolución de problemas

## Estado

Exploratorio

## Motivación

VSlices se está desarrollando actualmente como una suite de ingeniería de software. Se enfoca en preservar la continuidad entre:

* descubrimiento del dominio,
* documentación,
* razonamiento de diseño,
* arquitectura,
* implementación,
* validación,
* y evolución.

Sin embargo, mientras trabajaba en VSlices, empezó a aparecer un patrón más amplio.

Varias ideas usadas en VSlices no parecen exclusivas de la ingeniería de software. Parecen pertenecer a una forma más general de resolver problemas de manera progresiva:

* empezar desde el problema antes de asumir la solución,
* preservar el conocimiento antes de que se pierda,
* usar distintas modalidades de razonamiento dependiendo de la incertidumbre,
* documentar decisiones y tradeoffs,
* diseñar intervenciones acotadas,
* validar con la realidad,
* y evolucionar a partir de lo aprendido.

Esta teoría existe para explorar si VSlices puede entenderse como una implementación especializada en software de una metodología más amplia de resolución de problemas.

Esa metodología más amplia se llama tentativamente PResolver (pResolver). La teoría no dice que PResolver ya esté validado.

La teoría dice que VSlices puede proporcionar una especialización concreta desde la cual pueda abstraerse PResolver v0.1.

## Idea central

PResolver no debería inventarse primero como una metodología genérica.

Debería extraerse de VSlices después de observar qué patrones de VSlices siguen teniendo sentido cuando se eliminan los supuestos específicos de software. En esta teoría:

* VSlices es la especialización de ingeniería de software.
* PResolver es la abstracción más amplia.
* Domus Orbis y otros proyectos de software validan VSlices.
* Una práctica exitosa de VSlices puede revelar estructuras reutilizables de razonamiento, documentación y método.
* PResolver v0.1 documentaría esas estructuras reutilizables fuera del lenguaje específico de software.
* Experimentos posteriores de PResolver probarían dónde funcionan esas estructuras más allá del software.

La dirección importa. El camino práctico no es:

```text
Inventar PResolver
-> derivar VSlices
-> validar todo a la vez
```

El camino práctico es:

```text
Construir VSlices
-> validar VSlices en software
-> observar patrones recurrentes
-> abstraer estructuras no específicas de software
-> definir PResolver v0.1
-> probar PResolver en otros dominios
```

Esta teoría nos ayuda a ver VSlices de otra manera.

VSlices puede no ser solo una suite de ingeniería de software. También puede ser evidencia de que existe debajo de ella un modelo más amplio de resolución de problemas.

Esto no reemplaza a VSlices:

- No reduce la importancia de VSlices Framework.
- No implica que PResolver deba tener un framework.

Solo propone que las capas de razonamiento, documentación y método de VSlices pueden contener abstracciones útiles más allá del software.

## Vocabulario de trabajo

* PResolver: nombre tentativo para una metodología más amplia de resolución de problemas abstraída de VSlices.
* VSlices: especialización de ingeniería de software de la idea más amplia de PResolver.
* Specialization: aplicación concreta de principios más amplios a un dominio específico.
* Abstraction: el proceso de eliminar supuestos específicos de software mientras se preservan estructuras de razonamiento reutilizables.
* Problem Space: la situación más amplia que se explora antes de decidir qué tipo de solución se necesita.
* Intervention: una respuesta acotada a un problema; puede ser software, proceso, documentación, comunicación, política u otra forma de acción.
* Reasoning Modality: una forma de abordar la incertidumbre, como empezar desde el contexto, un problema o una pequeña intervención.
* Knowledge Preservation: documentación usada para mantener disponibles a lo largo del tiempo la comprensión relevante, las decisiones, los riesgos y el aprendizaje.
* Safeguard: un mecanismo proporcional que evita que una intervención cause daño evitable.
* Evidence Return: el acto de devolver los resultados de la validación a la comprensión y decisiones futuras.
* Software-Specific Pattern: un patrón que solo tiene sentido dentro de la ingeniería de software o la implementación.
* Generalizable Pattern: un patrón que sigue siendo útil después de eliminar los detalles específicos de software.

Estas definiciones son provisionales.

Pueden cambiar a medida que la teoría evoluciona.

## Ejemplos actuales

### Ejemplo 1: VSlices Method como especialización

VSlices Method conecta:

```text
domain discovery
-> documentation
-> design reasoning
-> architecture
-> implementation
-> validation
-> evolution
```

PResolver podría abstraer esto en:

```text
problem observation
-> knowledge preservation
-> resolution reasoning
-> intervention structure
-> intervention execution
-> validation
-> evolution
```

Los detalles específicos de software cambian.

La preocupación por la continuidad permanece.

Esto sugiere que VSlices Method puede ser una expresión especializada de un modelo más amplio de continuidad en la resolución de problemas.

### Ejemplo 2: Modalidades de diseño

VSlices Design usa modalidades como:

* Context-First,
* Problem-First,
* Slice-First.

Estas modalidades ayudan a decidir por dónde empezar según la incertidumbre de la situación actual de diseño de software.

PResolver podría abstraerlas en modalidades de razonamiento más amplias:

* Context-First: comenzar desde la situación más amplia cuando el problema no está claro.
* Problem-First: comenzar desde un dolor, necesidad o tensión visible.
* Intervention-First: comenzar desde una pequeña acción acotada cuando aprender de la realidad es más seguro que seguir analizando.

El lenguaje de software cambia.

La estructura del razonamiento permanece.

### Ejemplo 3: Documentación progresiva

VSlices Docs Standard trata la documentación como diseño vivo.

Los documentos preservan conocimiento que apoya la arquitectura, la implementación, la validación y la evolución.

PResolver podría generalizar esta idea.

La documentación preservaría conocimiento de resolución de problemas:

* lo que se observó,
* lo que se enmarcó como el problema,
* qué supuestos existen,
* qué intervención se seleccionó,
* qué tradeoffs se aceptaron,
* qué salvaguardas fueron necesarias,
* qué se validó,
* y qué debería evolucionar.

El objetivo no es más documentación. Es continuidad.

### Ejemplo 4: Daño mayoritario y salvaguardas minoritarias

Un patrón recurrente de resolución de problemas puede expresarse como:

> Resolver el daño dominante sin borrar daños reales menores.

En software, esto puede aparecer como:

* optimizar el caso de uso principal,
* identificar errores esperados,
* identificar casos de abuso,
* agregar salvaguardas,
* evitar sobrediseñar casos raros,
* pero no ignorar riesgos reales.

Fuera del software, el mismo patrón puede aplicar a problemas sociales, organizacionales, de políticas o de procesos.

Esto sugiere que algunos patrones de razonamiento de VSlices pueden tener un uso más amplio.

## Posible aplicación a VSlices

Esta teoría puede influir en VSlices aclarando su posición.

VSlices puede seguir enfocada en la ingeniería de software mientras se entiende como una especialización de una metodología más amplia.

Esto puede ayudar a explicar por qué VSlices empieza antes del código. VSlices no solo se ocupa de la implementación. Se ocupa de preservar la continuidad desde la comprensión hasta la implementación.

Si PResolver se convierte en la abstracción más amplia, VSlices puede seguir siendo concreta:

* VSlices Design sigue siendo razonamiento de diseño orientado al software.
* VSlices Docs Standard sigue siendo preservación de conocimiento orientada al software.
* VSlices Method sigue siendo continuidad del ciclo de vida del software.
* VSlices Framework sigue siendo la capa de implementación en .NET.

Esta teoría no debería cambiar VSlices Framework.

VSlices Framework existe porque VSlices tiene un objetivo concreto de implementación: los sistemas de software.

PResolver no debería heredar ese objetivo. No debería existir un PResolver Framework en esta teoría.

Como máximo, futuras herramientas de PResolver o más especializaciones podrían aparecer si casos de uso repetidos fuera del software lo justifican.

## Posible aplicación a PResolver

Esta teoría puede definir el primer camino hacia PResolver v0.1.

PResolver v0.1 no afirmaría ser una metodología universal validada. Documentaría la primera abstracción extraída de VSlices. Su suite inicial podría incluir:

* PResolver Method
* PResolver Design
* PResolver Docs Standard

### PResolver Method

PResolver Method describiría cómo pasar de situaciones poco claras a intervenciones responsables. Conectaría:

* observation
* framing
* reasoning
* documentation
* intervention
* validation
* evolution

No prescribiría un proceso rígido. Ayudaría a seleccionar la modalidad de razonamiento correcta según la incertidumbre.

### PResolver Design

PResolver Design describiría modalidades de razonamiento y herramientas de modelado para entender problem spaces. Ayudaría a responder:

* ¿Qué tipo de incertidumbre existe?
* ¿Deberíamos empezar por el contexto?
* ¿Deberíamos enfocarnos en un problema específico?
* ¿Deberíamos ejecutar una pequeña intervención?
* ¿Qué riesgos o salvaguardas importan?
* ¿Qué evidencia nos ayudaría a decidir?

### PResolver Docs Standard

PResolver Docs Standard definiría estructuras de documentación para preservar conocimiento de resolución de problemas.

Podría incluir documentos como:

* Shared Vocabulary
* Situation Context
* Problem Statement
* Intervention Proposal
* Safeguard Note
* Decision Record
* Validation Note
* Support Note.

Estos documentos no serían obligatorios por defecto.

Existirían solo cuando preserven conocimiento útil.

## Riesgos

Esta teoría puede volverse dañina si hace que VSlices sea demasiado abstracta.

VSlices no debe perder su identidad como suite de ingeniería de software. PResolver no debe robarle foco a VSlices mientras VSlices aún se está validando.

Esta teoría también puede volverse engañosa si asume que todo concepto de VSlices puede generalizarse.

Algunos conceptos pertenecen específicamente a la ingeniería de software. Forzarlos dentro de PResolver crearía abstracciones débiles.

La teoría también puede volverse demasiado amplia. Si PResolver intenta resolver todo tipo de problemas, puede volverse vaga e inútil.

Otro riesgo es la productización prematura.

Crear una suite completa de PResolver demasiado pronto puede producir documentación, branding y estructura antes de que existan suficientes ejemplos. El primer paso correcto es la abstracción, no la expansión.

PResolver v0.1 debería seguir siendo pequeña. Debe capturar solo los patrones que sobreviven a la abstracción de VSlices.

## Estado de validación

No existe validación directa para PResolver en este momento.

VSlices se está validando como una suite de ingeniería de software.

Si VSlices funciona, puede proporcionar evidencia de que algunas de sus estructuras de razonamiento, documentación y método pueden abstraerse en PResolver v0.1.

Eso no valida PResolver completamente.

Solo valida una especialización.

PResolver en sí deberá probarse más adelante en otros dominios.

El modelo de validación actual es:

```text
VSlices valida una especialización de software.
PResolver abstrae la estructura reutilizable.
Otros dominios prueban la abstracción.
```

Este documento representa solo una hipótesis exploratoria.
