# Context Guides

Context Guides explain how VSlices Method can be applied in different working contexts.

They do not define mandatory adoption paths. They help teams understand the situation they are entering, identify the main risks, and choose a useful starting point without forcing the same process everywhere.

## Purpose

A working context changes how Method should be applied. Starting from a non-digitalized scenario is not the same as joining an existing software project.

Introducing Method into ongoing work is not the same as using it from the beginning. Context Guides help answer questions such as:

* What kind of situation are we entering?
* What uncertainty dominates this context?
* What risks should we avoid?
* Which VSlices Design modality may be useful first?
* What knowledge should be preserved early?
* How can Method help without creating unnecessary ceremony?

## How to use these guides

Use a Context Guide as an orientation aid. Do not treat it as a checklist.

A guide may suggest useful starting points, common risks, and possible documentation support, but the team should still decide based on the real context.

The goal is not to apply Method completely. It is to preserve enough continuity for the next responsible decision.

## Available guides

* **Non-digitalized scenario** explains how to start when work is not yet represented by software.
* **Digitalized scenario** explains how to work when existing software already represents part of the business.
* **Joining an existing project** explains how to enter ongoing work without breaking continuity.
* **Introducing Method** explains how to introduce VSlices Method through small useful continuity seams.

## Guiding principle

Context Guides follow the same principle as VSlices Method:

> Use the smallest useful working structure that preserves the knowledge future work depends on.

A guide is useful only when it helps the team understand where they are, what matters now, and how to move forward without losing intent.
