# VSlices Docs Standard

VSlices Docs Standard defines reusable documentation structures for preserving knowledge across the lifecycle of a software system.

Its purpose is to help teams keep domain understanding, documentation, design reasoning, decisions, implementation behavior, validation, and evolution connected.

VSlices Docs Standard does not exist to make teams produce more documents. It exists to help teams preserve the knowledge that software decisions depend on.

## What it provides

VSlices Docs Standard provides:

- __principles__: the ideas that guide how documentation should be used.
- __document modeling__: how notes, documents, references, states, and relationships work.
- __taxonomy__: how documents are grouped by the kind of knowledge they preserve.
- __continuity paths__: how knowledge moves between documents, decisions, implementation, and learning.
- __anti-bureaucracy rules__: how to avoid turning documentation into ceremony.
- __document structures__: reusable formats for common kinds of engineering knowledge.

The standard is intentionally progressive. A team should use the smallest useful structure that preserves the knowledge future work depends on.

## What it is not

VSlices Docs Standard is not a complete methodology.

- It does not define the full workflow for running an iteration.
- It does not require every document to be created.
- It does not define meetings, roles, stage gates, or delivery rituals.

Those concerns belong to VSlices Method. Docs Standard defines the instruments and Method defines how those instruments are used during real work.

## How to use it

Start from the knowledge you need to preserve.

* **Use a Support Note** when knowledge is early, uncertain, local, or temporary.
* **Use a Domain Vocabulary** when language may affect understanding, behavior, naming, or boundaries.
* **Use a Context Document** when the team needs to preserve where work happens.
* **Use a Process Document** when responsibilities, coordination, workflows, or operational rules matter.
* **Use a Use Case Document** when behavior needs explicit meaning, consequences, validations, or expected errors.
* **Use a Capability Document** when several behaviors or processes depend on a stable ability.
* **Use a Decision Record** when a choice has meaningful tradeoffs, consequences, or future impact.
* **Use a Validation Note** when evidence changes or confirms knowledge that future work depends on.

Do not create a document because the standard contains it. Create it because it preserves useful knowledge.

## Relationship with the VSlices Suite

VSlices Docs Standard is one product inside VSlices Suite.

* **VSlices Design** defines how teams reason about design.
* **VSlices Docs Standard** defines how teams preserve different kinds of knowledge.
* **VSlices Method** defines how teams apply design and documentation during real work.
* **VSlices Framework** reflects that knowledge in implementation.

Docs Standard is independent from VSlices Framework. A team can use these documents with any technology, architecture, language, or delivery process.
